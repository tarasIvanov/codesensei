# Contract: GitHub Reviews API Payload + Mapping

This contract specifies (a) the exact outbound GitHub request shape and (b) the deterministic mapping from a `ReviewResult` to that shape.

## Outbound endpoint

```
POST https://api.github.com/repos/{owner}/{repo}/pulls/{number}/reviews
Host:                api.github.com
Authorization:       Bearer <GITHUB_TOKEN from app_settings>
Accept:              application/vnd.github+json
X-GitHub-Api-Version: 2022-11-28
User-Agent:          codesensei/0.0
Content-Type:        application/json
```

`owner`, `repo`, `number` derive from `pr_url` via the existing regex.

## Outbound request body

```json
{
  "event": "<COMMENT|REQUEST_CHANGES|APPROVE>",
  "body":  "<top-level markdown body>",
  "comments": [
    {"path": "src/auth.py", "side": "RIGHT", "line": 42, "body": "**blocker**: SQL built with f-string.\n\n_Suggestion_: Use parameterised query."}
  ]
}
```

- `event` — passed through verbatim from the request.
- `body` — see "Body composition" below.
- `comments` — see "Comment mapping" below. Empty array is legal (body-only review).

## Body composition

The top-level review body is composed in this order:

1. **Verdict line** — `Verdict: <UPPERCASE_VERDICT> — <N blocker>, <N major>, <N minor>, <N nit>` (severity counts omitted if zero across all four categories). Always present.
2. **Blank line.**
3. **Provider attribution** — `Generated by CodeSensei via {review_result.provider} in {review_result.elapsed_ms} ms.` Always present.
4. **Blank line.**
5. **Overflow / locationless block** — present *only* when at least one finding is going to be rendered into the body (locationless or above the 50-cap). Header line: `### Findings without inline location` (locationless only), or `### Additional findings (beyond the 50-comment cap)` (overflow), or both headers in sequence when both groups have entries. Each finding is one bullet:
   - `- **{severity}**: {message}` for locationless.
   - `- **{severity}** at `{file}:{line}`: {message}` for cap-overflow.
   - If `suggestion` is present, append `  ` (two-space line break) then `  _Suggestion_: {suggestion}` on the next line.

## Comment mapping

```python
def to_inline_comment(finding) -> dict:
    return {
        "path": finding.file,
        "side": "RIGHT",
        "line": finding.line,
        "body": _render_body(finding),
    }

def _render_body(finding) -> str:
    head = f"**{finding.severity}**: {finding.message}"
    if finding.suggestion:
        return f"{head}\n\n_Suggestion_: {finding.suggestion}"
    return head
```

A finding qualifies for inline rendering iff `finding.file` is a non-empty string AND `finding.line` is an int ≥ 1. Both conditions are checked independently (a finding with `file == ""` and a finding with `line is None` both demote to body).

## Selection order

1. Filter findings into two pools by predicate above: `inline_pool` (file + line) and `body_pool` (everything else).
2. Take the *first 50* findings from `inline_pool` (preserving original order — the LLM already orders by severity in the existing parser). These become `comments[]`.
3. Move the *remainder* of `inline_pool` into the body overflow list, preserving order.
4. Render `body` per "Body composition" above.

Result: every finding appears exactly once in the posted review.

## Severity text

`severity` is rendered using the existing `Severity` StrEnum values verbatim (`blocker`, `major`, `minor`, `nit`) — lower-case. Feature 003's `Finding` model carries no `category` field (the LLM-output JSON schema in `prompt.py` does not request one), so the markdown body shows severity only. If `Finding` ever gains a `category` field, this contract must be updated and the mapper must render it after the severity.

## Response handling

GitHub success response (200) fields we consume:

- `id` → `review_id`
- `html_url` → `html_url`
- *Everything else* is ignored (Pydantic `extra="ignore"`).

GitHub error responses are translated by status code per `contracts/api_post_review.md`. The translator inspects the JSON `message` and `errors[]` fields on 422 to distinguish the comment-position case (triggers the body-only fallback) from a structural rejection (skips the fallback).

## Position-error detection (for 422 fallback)

The body-only retry runs iff GitHub's 422 response body satisfies any of:

- `errors[]` contains an entry with `resource == "PullRequestReviewComment"` AND `field` matches `^(path|line|position|start_line)$`.
- The top-level `message` (lower-cased) contains the substring `"not part of the pull request diff"`.

Otherwise the 422 is treated as a structural rejection and surfaces as `github_review_rejected` without a retry.

## Timeouts and retries

- httpx timeout = 15 s per call.
- No automatic retry on 5xx / network errors. Caller (the frontend) re-submits via the retry button if the error envelope has `retryable=true`.
- The 422-fallback retry is *not* counted as a retry of the user's action — it is part of one logical post. `attempted_calls` distinguishes them for diagnostics.
