# Contract: `/review` "Post to GitHub" Panel UI

## Visibility predicate

The `PostToGitHubPanel` mounts on the `/review` page iff **both** of these are true:

1. `result` (the rendered `ReviewResult`) is non-null. The panel never appears alongside the empty / loading / error states of the review itself.
2. The review was generated by submitting a PR URL — i.e. the page's local state still carries the `pr_url` that produced `result`. Reviews generated from a pasted raw diff do not have a target PR and the panel must stay hidden.

When the predicate flips back to false (user clears the form, runs a new review from a different shape of input, etc.) the panel unmounts. Its internal state — including a successful post receipt — is discarded.

## Initial state

| Control | Initial value |
|---------|---------------|
| Event radio | `COMMENT` if `verdict == "comment"`, `REQUEST_CHANGES` if `verdict == "request_changes"`, `APPROVE` if `verdict == "approve"`. The reviewer may change this freely before clicking submit. |
| Submit button | Enabled, label `Post to GitHub`. |
| Error banner | Hidden. |
| Success state | Hidden. |

## Transitions and rules

- The IN-FLIGHT state must persist until the request resolves. Double-click during IN-FLIGHT is suppressed by the `inFlight` ref disabling the button.
- POSTED is **terminal** for the lifetime of this panel instance. Re-mounting (a new review result) creates a fresh instance with cleared state.
- ERROR banner content is derived from the typed `PostReviewError`:
  - `invalid_input` — text: "Invalid request — re-run the review and try again." No retry. (Should be unreachable from the UI; defensive.)
  - `settings_locked` — text: "GitHub token not configured." with a `<RouterLink to="/settings">` styled as a button. No retry.
  - `github_auth_failed` — text: the server message verbatim (it names the missing permission). No retry.
  - `github_pr_not_found` — text: "GitHub could not find this PR — check the URL." No retry.
  - `github_review_rejected` — text: "GitHub refused this review:" followed by `<pre>` of the server message. No retry.
  - `github_api_unavailable` — text: "GitHub is unreachable." Retry button.
  - `github_rate_limited` — text: "GitHub rate limit hit — retry in {retry_after_seconds}s." Retry button is *disabled* with a live countdown until the deadline, then re-enabled.
  - `internal` — text: "Unexpected server error." Retry button (defensive — a transient backend hiccup is the legitimate case).

## Success state

A single line: `Posted ✓ — <a href="{html_url}" target="_blank" rel="noopener">View on GitHub</a>`. Below it: `Inline comments attached: {comment_count}`. No further controls. The link must open in a new tab so navigating back is one browser-back.

## Accessibility

- Radio group has an accessible `<legend>` "Review action".
- Submit button has `aria-busy="true"` during IN-FLIGHT.
- Error banner has `role="alert"` so screen readers announce it on appearance.
- Retry button is keyboard-focusable; the countdown updates `aria-live="polite"`.

## Frontend wire

The panel calls one API function:

```ts
postReview(input: PostReviewInput): Promise<PostedReviewReceipt>
```

…which is defined in `frontend/src/api/posting.ts` and throws `PostReviewError` on non-2xx responses. The panel does not parse error envelopes itself; the typed client does.
